## Decisions Made This Session

1. **Read token usage from JSONL transcript, not count tool calls (D-009)**
   - Why: Tool calls are a garbage proxy — one call = 50 or 50,000 tokens
   - Alternatives rejected: Tool-call counting, token estimation heuristics, time-based warnings

2. **Check tokens every 5th tool call (D-010)**
   - Why: 23ms per check is fine but unnecessary every call; 5th gives adequate granularity
   - Alternatives rejected: Every call, every 10th

## Things Tried That Failed

1. **grep-based token extraction**: `grep '"input_tokens"'` matches command text in JSONL (our own commands referencing "input_tokens"), not just API usage data. Had to switch to python3 JSON parsing.
2. **tail -c 50000**: Too small — recent tool results (file writes) dominated the last 50KB, pushing usage data out of range. Increased to 500KB.

## Open Questions / Blockers

1. **Warning visibility**: Do stderr warnings from the hook actually appear in the Claude Code conversation? Still untested — need to reach 100K tokens naturally.
2. **Threshold calibration**: 100K/130K/150K based on observed ~160K compaction. May need adjustment with more data.
3. **Portability**: Token reading is Claude Code-specific (reads its JSONL format). Other providers would need different implementations.
